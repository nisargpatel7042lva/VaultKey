# SBF build error E0152 (duplicate `core` / `sized`)

If `anchor build` / `cargo build-sbf` fails with errors like:

```text
error[E0152]: duplicate lang item in crate `core` ... `sized`
  first definition in .../target/.../libcore-....rmeta
  second definition in .../platform-tools/.../libcore-....rlib
```

Rust is loading **two different copies** of the `core` library for the `sbpf-solana-solana` target. This often appears with **Agave / Solana CLI 3.x** + **platform-tools v1.52** on some setups (WSL2 included).

## Things to try (in order)

### 1. Refresh platform-tools

```bash
cd programs/vault
cargo build-sbf --force-tools-install --tools-version v1.52
```

Then from repo root:

```bash
rm -rf target/sbpf-solana-solana programs/vault/target/sbpf-solana-solana
anchor build -p vault --no-idl
```

### 2. Use Solana **2.0.x** CLI for building (recommended if 1 keeps failing)

Install a **2.0** release (or another LTS you use for known-good Anchor builds), put **that** `cargo-build-sbf` on `PATH`, and rebuild:

```bash
solana-install init 2.0.25   # example; pick a 2.0.x you prefer
hash -r
rm -rf target/sbpf-solana-solana
anchor build -p vault --no-idl
```

Deploy with **either** CLI; program binaries are compatible with devnet as long as the BPF/SBF target matches what validators expect.

### 3. Build elsewhere

- Another Linux machine or **GitHub Actions** with a pinned Solana + Anchor image.
- Copy `target/deploy/vault.so` (or `programs/vault/target/deploy/vault.so`) back and run:

  ```bash
  solana program deploy path/to/vault.so \
    --program-id target/deploy/vault-keypair.json \
    -u devnet
  ```

### 4. Repo settings we already relaxed

Root `Cargo.toml` no longer uses `lto = "fat"` / `codegen-units = 1` on the workspace `release` profile, which can worsen fragile SBF builds.

---

After you have `vault.so`, deploy and run `pnpm init-vault` as in `DEPLOY_BACKEND.md`.
