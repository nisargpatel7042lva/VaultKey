# SBF build error E0152 (duplicate `core` / `sized`)

If `anchor build -p vault` fails with:

```text
error[E0152]: duplicate lang item in crate `core` ... `sized`
  ... libcore-....rmeta
  ... libcore-....rlib
```

Rust is loading **two different `libcore` builds** for `sbpf-solana-solana`. This shows up often on **WSL2** with **Agave / Solana 3.1.x** when the host `cargo-build-sbf` / `~/.cache/solana` layout gets inconsistent.

## Fix that works (recommended): Docker build

From the repo root (requires **Docker**):

```bash
pnpm run build:vault:sbf
```

Same as:

```bash
bash scripts/build-vault-sbf.sh
```

This runs `Dockerfile.vault-sbf` (clean Ubuntu 22.04, Agave **v3.0.10** for `cargo-build-sbf`, Anchor **0.32.1**) and copies `vault.so` to **`target/deploy/vault.so`**.

Then deploy:

```bash
solana program deploy target/deploy/vault.so \
  --program-id target/deploy/vault-keypair.json \
  -u devnet
pnpm init-vault
```

## CI artifact

GitHub Actions workflow **“Build vault (SBF, Docker)”** uploads **`vault.so`** as a build artifact. Use your local **`target/deploy/vault-keypair.json`** (or generate one) for `--program-id` when deploying.

## Notes

- The repo **`.dockerignore`** keeps the Docker build context small (excludes `node_modules`, `target`, `.git`). First image build still downloads toolchains inside the container and can take several minutes.
- `solana-install` is not required; your existing **Agave 3.1.x** CLI can stay on `PATH` for `deploy` / RPC.
- Root `Cargo.toml` avoids aggressive workspace `lto = "fat"` on `release` (helps some fragile SBF builds).
- Do **not** commit a stray `programs/vault/Cargo.lock` (workspace should use the repo-root `Cargo.lock` only).
